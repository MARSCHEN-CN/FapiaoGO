# 入口差异审计报告

## 核心发现：解析器相同，但响应链不同

### 两条入口的调用链对比

```
/parse_invoice                          /import/batch
    │                                       │
    ▼                                       ▼
_run_parse_offthread()                  ImportBatchManager
    │                                       │
    ├─ ocr_pool_task.run_parse()            ▼
    │      │                          ParseJobManager._execute_job()
    │      ▼                               │
    │  parse_invoice_service()              ▼
    │  (惰性导入)                      job._parse_func(*args, **kwargs)
    │                                       │
    └─ _parse_sync() (回退)                  ▼
           │                          parse_invoice_service()
           ▼                          (已提前 import，直接传引用)
      parse_invoice_service()
```

**结论：两个入口最终都调用 `parse_invoice_service(file_bytes, filename, auto_orient, enable_auto_ocr, skip_db_write=True)`。**

### 但响应构建不同

| | `/parse_invoice` | `/import/batch` |
|---|---|---|
| 前端传 `mode` | `mode=batch`（由 prepareSingleRequest 硬编码） | ❌ 不传 |
| `build_response(..., mode='batch')` | `include_preview=False, include_raw_text=False` | 走默认值，不同 |
| 响应 JSON | 直接返回给 fetch，前端 `createParseResult` 消费 | 经 `set_ocr_cache` 缓存，`_on_job_done` 回调，`get_batch_results` 组装 |
| 最终消费 | `runFallbackParseTask` → `createParseResult` → `mapParseResultToFileUpdate` | `consumeParseResult` → `ImportSessionStore` → hydration |

### `mode='batch'` 的实际影响

在 `response_builder.py:42-45`：

```python
if mode == 'batch':
    if file_format != 'ofd':
        include_preview = False
    include_raw_text = False
```

这只影响返回中的 **preview_image** 和 **raw_text** 字段，不影响 `invoice_fields`（含 `gmfmc`）。

### 真正的分叉点在前端消费侧

```
单文件路径：parseResultMapper.mapParseResultToFileUpdate
批量路径：  consumeParseResult → ImportSessionStore + DocumentStore + hydration
```

这是 **两条不同的前端状态机**。同一份后端 JSON，两套映射逻辑，可能产生不同的 `failed_fields` 推断。

## 结论

- ✅ **解析器相同**：`parse_invoice_service()` 是单一事��源
- ✅ **文件内容相同**：`prepareSingleRequest` 通过 IPC read-file 读到的 `size: 25835` 与 batch 预读一致
- ⚠️ **响应构建差异**：`mode='batch'` 仅影响 preview/raw_text，不影响 invoice_fields
- ❓ **前端消费差异**：`mapParseResultToFileUpdate` vs `consumeParseResult` 可能是真正分叉点

若要进一步追踪，下一个对比目标是：

```
frontend/mappers/parseResultMapper.js  →  mapParseResultToFileUpdate
               VS
frontend/consumers/parseResultConsumer.js  →  consumeParseResult
```
