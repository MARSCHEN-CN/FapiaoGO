# 前端消费器对比审计结果

## 两条路径的数据流

```
后端 JSON（parse_invoice_service 输出）
    │
    ├── 单文件路径
    │   build_response(mode='batch')    ← 经过 response_builder 加工
    │   → resp.json()
    │   → createParseResult(data, name) ← 标准化 ParseResult
    │   → mapParseResultToFileUpdate(result, fileObj)
    │
    └── 批量路径
        SSE stream → batchResult.items  ← 未经过 response_builder
        → consumeParseResult(result, fileObj)
          → mapParseResultToFileUpdate(result, fileObj)  ← 同一 mapper!
```

## 关键发现

### 1. `consumeParseResult` 只是 `mapParseResultToFileUpdate` 的薄包装

```js
// parseResultConsumer.js:34
const update = mapParseResultToFileUpdate(result, fileObj)
```

**不做额外字段处理**，只写入 Store 和 Document 注册。

### 2. 批量 hydration 也调同一个 mapper

```js
// useFileOps.js:493-515
const hydrationResult = {
  status: 'parsed',
  fields: {
    invoiceType: item.invoiceType || '',
    invoiceNumber: item.invoiceNumber || '',
    amount: item.amount || '',
    invoiceDate: item.invoiceDate || '',
    failedFields: item.failedFields || [],
    invoiceFields: item.invoiceFields || null,
    ...
  },
  raw: {},
}
const update = mapParseResultToFileUpdate(hydrationResult, fileObj)
```

**注意**：batch hydration 构造的 `raw: {}` 是空对象！而单文件路径 `createParseResult` 的 `raw: data` 包含完整原始响应。

### 3. `mapParseResultToFileUpdate` 从 `raw` 取 fallback

```js
// parseResultMapper.js:30-31
invoiceType: fields.invoiceType || raw.invoice_type || '',
invoiceNumber: fields.invoiceNumber || raw.invoice_number || '',
```

当 `fields.*` 为空时，单文件路径能从 `raw` 兜底，批量路径 `raw={}` 则拿不到。

## 结论

**两个路径的消费链已经高度对齐**，都是同一个 mapper，同一个 ParseResult 模型。如果同一张票在两个入口解析结果不同，问题大概率不在消费器，而在：

1. **两边的 data 来源不同** — 单文件路径经过 `build_response()` 加工（clean_type / clean_amount / clean_date），批量路径走 `get_batch_results` 从 ocr_cache 读原始 dict
2. **mode='batch' 导致 preview_image="" 和 raw_text=""** — 但 gmfmc 不受影响
3. **文件本身** — 重命名后文件可能在磁盘上尚未完全就绪就被 read-file 读取

下一步如果要继续追，建议只做一件事：**在 `mapParseResultToFileUpdate` 入口处加一行 console.log，对比两个路径进入时的 `result.fields.invoiceFields.gmfmc` 值**。
